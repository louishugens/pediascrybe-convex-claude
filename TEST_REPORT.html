4 · Bug found & fixed — Password reset (P1)

FieldDetail
SymptomUser reported "trying to reset my password and it is failing."
Root causeApp called authClient.forgetPassword() → POST /api/auth/forget-password. Better Auth 1.4.9 renamed this route; the path now 404s (only the email-OTP plugin defines /forget-password/email-otp). The reset email was therefore never requested.
DiagnosisDirect endpoint probing: POST /api/auth/forget-password → 404; POST /api/auth/request-password-reset → 200 and logs confirmed the Resend email sent. Route definitions verified in node_modules/better-auth/dist/api/routes/reset-password.mjs.
FixOne line in app/(public)/reset/page.jsx:
- const { error } = await authClient.forgetPassword({
+ const { error } = await authClient.requestPasswordReset({
    email: values.email,
    redirectTo: `${window.location.origin}/reset/password`,
  })
Step 2 of the flow — setting the new password via authClient.resetPassword({newPassword, token}) → POST /reset-password — was already correct and unchanged. Verified end-to-end through the browser client; guarded by regression test e2e/reset.public.spec.ts.

8 · Full-feature dev testing round

Every authenticated doctor route (46 routes) was render-smoke tested, plus deep write-path flows. Run first against a production build (next start), then re-checked on the dev server.

Key learnings about the test environment

  • Turbopack dev cold-compile is very slow (30–76s per route, single-queued) — route-by-route smoke against dev times out. A production build serves every route in <1s and is the right target for broad smoke.
  • Vercel BotID blocks headless login in production builds (returns HUMAN only in dev). Authenticated prod testing must reuse an existing session (E2E_REUSE_SESSION=1 + a saved e2e/.auth/doctor.json).
  • Client-rendered pages (useQuery) can intermittently show the error boundary in the harness when the server-side session reports a transient "Not authenticated" — a JWT-refresh timing artifact, not an app defect (every route returns 200; identical pages pass; resolves on retry).

Bug #2 found & fixed — add-recommendation 500 (P1)

FieldDetail
Symptom/user/patients/[patientId]/[appointmentId]/add-recommendation returned HTTP 500 (dev and prod).
Root causeThe server component AddRecommendationContainer had a "use cache" directive but then called fetchAuthQuery, which reads headers(). Next.js 16 Cache Components forbids dynamic data sources (headers) inside a "use cache" scope → hard 500. The sibling add-exams page is identical but (correctly) has no "use cache".
FixRemoved the stray 'use cache' from add-recommendation/page.tsx. Route now returns 200. Confirmed via dev re-check.
