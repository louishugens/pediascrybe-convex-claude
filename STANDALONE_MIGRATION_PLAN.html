Standalone Client Migration Plan

Turn this copy of PediaScrybe (Next.js 16 + Convex + Better Auth) into a private, single-client deployment: strip all Stripe billing and subscription gating so every feature is unlocked, block new public signups, and migrate every user (with their existing passwords) plus all clinical data from the client's older Supabase build into the fresh Convex project.

Next.js 16 · App Router Convex backend Better Auth (scrypt) ✓ New Convex project linked ✓ Full Postgres access to source Lazy bcrypt → scrypt passwords Remove Stripe + gating Block signups

0How it will work — the short version

Three independent workstreams, run in this order. Each is reversible until you cut over.

A · Ungate everything

The app is unlocked by centralizing the "do you have access?" checks so they all answer yes, unlimited. We don't rip gates out of dozens of call-sites — we make the handful of gate functions return an unlimited "institution-equivalent" plan. Then we remove only the things that actively reach out to Stripe (webhook route, billing cron, customer-creation on signup) and neutralize the billing UI.

B · Auth migration

Every Supabase user is recreated in Better Auth with their email, name, role and original bcrypt password hash. A custom verify() hook detects bcrypt vs scrypt at login: bcrypt users sign in with their existing password and are transparently re-hashed to scrypt. Public signup is turned off at the API and in the UI.

C · Data migration

A one-shot script reads the Supabase Postgres tables, transforms them (dates → epoch-ms, embedded meds/exams → standalone rows), and inserts them into Convex through internal mutations. Because Convex generates its own IDs, a migrationMap table remaps every foreign key as rows are inserted, parents before children.

End state
The client opens the app, logs in with the same email + password they had on the old Supabase system, and sees all their patients, appointments, prescriptions, labs, vaccines, reports and receipts — with every feature (AI, telehealth, WhatsApp ScrybeGPT, patient portal) fully enabled and no paywall, trial banner, or usage meter anywhere. Nobody new can self-register.
Supabase Postgres (old app) → migrate-from-supabase.ts (reads + transforms) → Convex internal mutations → Better Auth users + appUsers + doctors + patients + … → New Convex project

4Workstream B — Auth: import users + lazy password migration

4.1 The password problem & the fix

Supabase Auth (GoTrue) stores bcrypt hashes ($2a$…) in auth.users.encrypted_password. Better Auth stores scrypt hashes in the format salt:hexkey in account.password. They are not interchangeable — but Better Auth lets us supply a custom verifier. We register one that handles both formats:

// convex/auth.ts → createAuthOptions(...)
import bcrypt from "bcryptjs";
import { hashPassword, verifyPassword } from "better-auth/crypto"; // scrypt default

emailAndPassword: {
  enabled: true,
  disableSignUp: true,            // ← closes public registration (§5)
  requireEmailVerification: false, // migrated users are imported pre-verified
  sendResetPassword: async ({ user, url }) => { /* unchanged */ },
  password: {
    hash: (pw) => hashPassword(pw),       // new/rehashed passwords → scrypt
    verify: async ({ password, hash }) => {
      if (hash.startsWith("$2")) {       // bcrypt from Supabase
        return bcrypt.compare(password, hash);
      }
      return verifyPassword({ password, hash }); // native scrypt
    },
  },
},
Result
A migrated user types their old password → verify() sees a $2… hash → bcrypt match → logged in. Native (post-reset / new) passwords use scrypt. Zero friction, no mass password reset. Add bcryptjs to convex/ deps (pure-JS, runs in Convex's runtime).

4.2 Rehash-on-login (optional, recommended)

To migrate each user off bcrypt permanently, re-hash on first successful login. Better Auth's verify() is pure, so do the rehash in a sign-in lifecycle hook (Better Auth databaseHooks / an after sign-in callback) that, when the stored hash is bcrypt, computes hashPassword(plaintext) and updates account.password. If wiring that hook is fiddly in the Convex adapter, it's safe to skip — the dual verifier works indefinitely; bcrypt simply lingers until the user next changes their password.

4.3 Creating the users during import

Each Supabase user becomes a Better Auth user + a credential account carrying the bcrypt hash. Do this through the Better Auth component adapter inside a Convex internal mutation so the onCreate trigger fires and auto-builds appUsers/doctors:

// convex/migration.ts
export const importUser = internalMutation({
  args: { email: v.string(), name: v.string(), role: v.string(),
          firstName: v.optional(v.string()), lastName: v.optional(v.string()),
          bcryptHash: v.string(), legacyUserId: v.string() },
  handler: async (ctx, a) => {
    const adapter = authComponent.adapter(ctx);
    // 1) create the Better Auth user (fires onCreate → appUsers + doctors/relink)
    const user = await adapter.create({ model: "user", data: {
      email: a.email, name: a.name, emailVerified: true,
      role: a.role, firstName: a.firstName, lastName: a.lastName,
      createdAt: Date.now(), updatedAt: Date.now(),
    }});
    // 2) attach the credential account with the original bcrypt hash
    await adapter.create({ model: "account", data: {
      userId: user.id, accountId: user.id, providerId: "credential",
      password: a.bcryptHash, createdAt: Date.now(), updatedAt: Date.now(),
    }});
    // 3) record legacy→convex id mapping for FK remapping
    await ctx.db.insert("migrationMap", {
      entity: "user", legacyId: a.legacyUserId, convexId: user.id });
    return user.id;
  },
});

Note: exact adapter method names (create vs createUser) are confirmed against @convex-dev/better-auth@0.10.9 during implementation — the shape above matches its adapter surface. The onCreate trigger already relinks doctors by email, so importing a doctor user produces a bare doctors row that step 6.4 then enriches with the full profile.

Turn off side-effects during import
Set MIGRATION_MODE=1 (Convex env) and guard the welcome-email scheduling in the onCreate trigger on it, so importing N users doesn't send N welcome emails. The Stripe-customer scheduling is already removed in §3.2.

6Workstream C — Data migration (Supabase → Convex)

Because the source is an older version of this same app, tables line up closely. The work is: connect to Postgres, transform values to Convex conventions, and insert parents-before-children while remapping IDs.

6.1 The ID-remapping mechanism

Convex assigns its own _id on insert — you cannot carry Supabase UUIDs into _id. So we keep a translation table and resolve every foreign key through it.

// add to convex/schema.ts (temporary; drop after migration)
migrationMap: defineTable({
  entity: v.string(),     // "doctor" | "patient" | "appointment" | …
  legacyId: v.string(),   // the Supabase PK (uuid/int as text)
  convexId: v.string(),   // the new Convex _id
}).index("by_entity_legacy", ["entity", "legacyId"]),

Each import mutation writes its (entity, legacyId) → convexId row, and child imports call a resolve(entity, legacyId) helper to fill in doctorId, patientId, appointmentId, etc. Idempotent: re-running checks the map first and skips/updates.

6.2 Order of operations (parents before children)

1
Users (doctors first, then patients) → importUser (§4.3). Trigger builds appUsers + bare doctors. Map user + (by email) doctor.
2
Enrich doctor profiles — patch each doctors row (phone, address, spec, title, summary, experience, cost, duration, timezone, availability, flags) by matching on email.
3
Patients → resolve doctorId from map; insert; map patient legacy→new.
4
patientAccounts (parent ↔ patient links) → resolve parent authUserId + patientId.
5
Appointments → resolve doctorId+patientId; convert vitals/dates. If the old schema embedded medication[]/exams[] on the visit, split them out into prescriptions / labOrders (the repo already did this once via migrateEmbeddedClinical.ts — reuse that mapping logic).
6
Children of appointments/patients — prescriptions, labOrders + labResults, services, vaccins → doses → vaccinationRecords, reports, receipts, files, doctorImages. Resolve all FKs through the map.
7
Verify counts per table vs source, then run the test-user validation (§12).

6.3 The runner script

// scripts/migrate-from-supabase.ts  (run with: tsx)
import { Client } from "pg";
import { ConvexHttpClient } from "convex/browser";
import { internal } from "../convex/_generated/api";

const pg = new Client({ connectionString: process.env.SUPABASE_DB_URL });
const convex = new ConvexHttpClient(process.env.NEXT_PUBLIC_CONVEX_URL!);
convex.setAdminAuth(process.env.CONVEX_DEPLOY_KEY!); // admin → can call internal.*

// 1) users (join auth.users + your doctors/profiles table)
const { rows: users } = await pg.query(`
  select u.id, u.email, u.encrypted_password as hash,
         p.firstname, p.lastname, p.role
  from auth.users u join profiles p on p.user_id = u.id`);
for (const u of users) {
  await convex.mutation(internal.migration.importUser, {
    email: u.email, name: `${u.firstname} ${u.lastname}`.trim(),
    role: u.role ?? "doctor", firstName: u.firstname, lastName: u.lastname,
    bcryptHash: u.hash, legacyUserId: u.id });
}
// 2..6) patients → appointments → prescriptions → … (same pattern, FK via map)

The script does the SQL reads + value transforms; all writes go through versioned Convex internal.migration.* mutations so validation and idempotency live in one place. Use CONVEX_DEPLOY_KEY (already in your .env.local) for admin access to internal functions.
